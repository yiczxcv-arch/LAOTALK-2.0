# LAOTALK 2.0 PROJECT RULE

Version 1.0

---

# 목적

본 문서는 LAOTALK 2.0 프로젝트를 개발하는 모든 작업자가 반드시 따라야 하는 공통 규칙을 정의한다.

설계 문서(MASTER PLAN, BLUEPRINT, DESIGN SYSTEM)를 기준으로 일관된 품질을 유지하는 것을 목표로 한다.

---

# 1. 개발 원칙

1. 설계가 완료되기 전에는 코드를 작성하지 않는다.
2. 모든 기능은 문서를 기준으로 개발한다.
3. 추측하여 기능을 추가하지 않는다.
4. 변경이 필요한 경우 CHANGELOG에 먼저 기록한다.
5. 모든 작업은 모바일 기준으로 검토한다.

---

# 2. 작업 순서

1. MASTER_PLAN 검토
2. BLUEPRINT 검토
3. DESIGN_SYSTEM 검토
4. 기능 구현
5. 테스트
6. DAILY_REPORT 작성
7. CHANGELOG 기록

---

# 4. 파일 작성 규칙

- 의미 있는 파일명을 사용한다.
- 기능별로 파일을 분리한다.
- 중복 코드를 만들지 않는다.
- 공통 요소는 재사용 가능한 컴포넌트로 작성한다.

---

# 5. UI 구현 규칙

- DESIGN_SYSTEM을 그대로 따른다.
- 임의의 색상이나 폰트를 추가하지 않는다.
- 버튼과 카드 스타일을 통일한다.
- 모든 페이지에 예약 CTA를 유지한다.

---

# 6. 콘텐츠 규칙

- 문장은 간결하게 작성한다.
- 이미지와 영상을 우선한다.
- 과장되거나 확인되지 않은 정보를 게시하지 않는다.
- 모든 콘텐츠는 브랜드 톤을 유지한다.

---

# 7. 예약 기능 규칙

- 회원가입 기능 추가 금지
- 홈페이지 결제 기능 추가 금지
- 예약은 문의 접수 방식으로 운영
- 카카오 상담을 기본 연결 채널로 사용

---

# 8. 관리자 규칙

관리자는 다음 기능만 제공한다.

- 예약 관리
- 콘텐츠 관리
- 액티비티 관리
- 골프 관리
- 패키지 관리

권한이 없는 기능은 추가하지 않는다.

---

# 9. 성능 규칙

- 모바일 우선 최적화
- 빠른 로딩 유지
- 이미지 최적화
- 불필요한 라이브러리 사용 금지

---

# 10. 보안 규칙

- 개인정보는 최소한만 수집한다.
- 민감한 정보는 로그에 저장하지 않는다.
- 입력값 검증을 수행한다.

---

# 11. Git 작업 규칙

- 기능 단위로 작업한다.
- 의미 있는 커밋 메시지를 사용한다.
- 큰 변경은 문서 검토 후 진행한다.

예시

feat: activity page
fix: reservation form
docs: update blueprint

---

# 12. 테스트 규칙

확인 항목

- 모바일 화면
- 태블릿 화면
- PC 화면
- 링크 동작
- 예약 폼
- 관리자 기능

---

# 13. 금지 사항

- 설계와 다른 기능 추가
- 임의의 UI 변경
- 문서 없이 구조 변경
- 테스트 없이 배포
- 기존 원칙을 무시한 개발

---

# 14. 인수인계 규칙

매일 작업 종료 시 DAILY_REPORT를 작성한다.

보고 내용

- 오늘 작업
- 완료 기능
- 미완료 작업
- 문제점
- 다음 작업
- 특이사항

---

# 15. 변경 관리

설계 변경 시 순서를 반드시 따른다.

1. MASTER_PLAN 영향 확인
2. BLUEPRINT 수정
3. DESIGN_SYSTEM 수정(필요 시)
4. PROJECT_RULE 수정(필요 시)
5. CHANGELOG 기록
6. 개발 반영

---

# 16. 완료 기준

다음 조건을 모두 만족해야 개발 완료로 본다.

- 설계 문서와 일치
- 모바일 UX 확인
- 예약 흐름 정상 동작
- 관리자 기능 정상 동작
- DAILY_REPORT 작성 완료
- CHANGELOG 기록 완료

본 문서는 프로젝트 진행 중 항상 최신 상태로 유지한다.
